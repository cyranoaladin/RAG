# ADR-0001 : Migration de ChromaDB vers pgvector

| Méta-données | Valeur |
|--------------|--------|
| **Titre** | Migration de ChromaDB vers pgvector |
| **Statut** | Accepted |
| **Date** | 2026-02-22 |
| **Auteur** | Alaeddine BEN RHOUMA |
| **Impact** | Architecture, Performance, Maintenance |

## Contexte

Le projet rag-local utilise initialement ChromaDB comme base de données vectorielle pour le stockage des embeddings. Cependant, plusieurs limitations ont été identifiées :

1. **Scalabilité** : ChromaDB a des limitations en termes de volume de données et de concurrence
2. **Flexibilité des requêtes** : Les filtres metadata sont limités
3. **Écosystème** : Moins mature que PostgreSQL pour la production
4. **Hybrid Search** : Difficile d'implémenter une recherche hybride (dense + sparse)

## Forces en jeu

- **Performance** : Besoin de requêtes rapides même avec des millions de documents
- **Flexibilité** : Pouvoir filtrer par tenant, domaine, tags, dates
- **Maintenance** : Outils de backup/restore matures
- **Observabilité** : Métriques et monitoring avancés
- **Coût** : Solution open-source, pas de dépendance cloud

## Décision

Nous décidons de migrer de ChromaDB vers **pgvector** (extension PostgreSQL) pour les raisons suivantes :

1. **PostgreSQL + pgvector** combine stockage relationnel et vectoriel
2. **Index HNSW** pour des requêtes ANN (Approximate Nearest Neighbor) performantes
3. **Requête hybride** native : combinaison de filtres SQL et recherche vectorielle
4. **Écosystème mature** : backup, replication, monitoring, pooling de connexions
5. **Multi-tenancy** : isolation par schema ou row-level security

### Schéma de données

```sql
-- Documents (métadonnées)
CREATE TABLE rag_documents (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant VARCHAR(50) NOT NULL,
    source_type VARCHAR(50) NOT NULL,
    source_path TEXT NOT NULL,
    title TEXT,
    file_hash VARCHAR(64) NOT NULL,
    embed_model VARCHAR(100),
    embed_dim INTEGER,
    label VARCHAR(100),
    mime_type VARCHAR(100),
    char_count INTEGER,
    metadata JSONB DEFAULT '{}',
    created_at TIMESTAMPTZ DEFAULT NOW(),
    updated_at TIMESTAMPTZ DEFAULT NOW(),
    UNIQUE(source_path, tenant)
);

-- Chunks avec embeddings
CREATE TABLE rag_chunks (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    document_id UUID REFERENCES rag_documents(id) ON DELETE CASCADE,
    tenant VARCHAR(50) NOT NULL,
    chunk_index INTEGER NOT NULL,
    text TEXT NOT NULL,
    embedding VECTOR(768),  -- nomic-embed-text
    char_start INTEGER,
    char_end INTEGER,
    page_number INTEGER,
    metadata JSONB DEFAULT '{}',
    created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Index vectoriel HNSW
CREATE INDEX ON rag_chunks USING hnsw (embedding vector_cosine_ops);
```

## Conséquences

### Positives

- ✅ **Performance** : Requêtes 2-5x plus rapides avec index HNSW
- ✅ **Flexibilité** : Filtres SQL complexes + recherche vectorielle
- ✅ **Hybrid Search** : Combinaison dense (vector) + sparse (BM25/tsvector)
- ✅ **Reliability** : PostgreSQL est éprouvé en production
- ✅ **Backup/Restore** : Outils matures (pg_dump, WAL, replication)
- ✅ **Multi-tenancy** : Isolation native avec RLS (Row Level Security)

### Négatives

- ⚠️ **Migration** : Effort de migration des données ChromaDB → pgvector
- ⚠️ **Complexité** : Architecture plus complexe (Celery, Redis, asyncpg)
- ⚠️ **Mémoire** : PostgreSQL nécessite plus de RAM que ChromaDB
- ⚠️ **Breaking Change** : Changement d'API pour les clients existants

### Neutres

- ➖ **Dépendance** : Ajout de PostgreSQL comme dépendance
- ➖ **Monitoring** : Nouvelles métriques à surveiller (connections, mémoire)

## Migration

La migration se fait en plusieurs étapes :

1. **Phase 1** : Coexistence ChromaDB + pgvector
2. **Phase 2** : Migration des données (script dédié)
3. **Phase 3** : Bascule du trafic
4. **Phase 4** : Décommissionnement de ChromaDB

Voir `MIGRATION_PLAN.md` pour les détails.
